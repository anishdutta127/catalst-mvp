## Section 2: Code Quality Review — 3 issues found

### CQ #1: (confidence: 9/10) journeyStore.ts — SSR-unsafe localStorage access at module scope

Lines 289-301 access `localStorage` at module load time with `if (typeof window !== 'undefined')`. This runs during SSR in Next.js App Router, causing hydration mismatches (server generates one sessionId, client generates another).

**Fix:** Move session initialization into a `useEffect` in the root layout or a provider component. Use Zustand `persist` middleware (which handles SSR correctly) instead of manual localStorage.

### CQ #2: (confidence: 8/10) uiStore.ts — messageCounter is a module-level mutable variable

`let messageCounter = 0` at line 77 is shared across all renders. In React 18 concurrent mode or during SSR + hydration, this can produce duplicate IDs. Not catastrophic but fragile.

**Fix:** Use `crypto.randomUUID()` for message IDs (already used for sessionId). No counter needed.

### CQ #3: (confidence: 9/10) components/characters/CedricBubble.tsx + PipBubble.tsx — onComplete closure issue

**Prior learning applied: framer-motion-onComplete-closures** (confidence 9/10, from 2026-04-14)

These components use word-by-word streaming with Framer Motion. If `onComplete` is an inline arrow function, React recreates it every render, potentially restarting the animation. Must use `useRef` for the callback and a stable `useCallback` wrapper.

## Section 4: Performance Review — 2 issues found

### Perf #1: (confidence: 8/10) ideas.json is 260 entries loaded client-side

`content/ideas.json` is loaded into the browser. At ~50 fields per idea, this is likely 200-400KB of JSON. The scoring engine needs all 260 ideas to run.

**Options:**
- A) Pre-compute during build (Next.js `generateStaticParams` or a build script that scores common profiles and caches top matches). Over-engineered for MVP.
- B) Accept it. 300KB gzipped is ~50KB. Loads once, cached by browser. The scoring engine runs in <100ms on modern devices. Not a real problem for MVP.

**Recommendation:** B. Accept it. Premature optimization.

### Perf #2: (confidence: 7/10) Background images (12 PNGs) could be large

12 Midjourney-generated backgrounds at full resolution could be 5-10MB total. Need to be optimized.

**Fix:** Use Next.js Image component with `priority` on current screen, `loading="lazy"` on others. Convert to WebP. Consider preloading next screen's background during current screen.

### OV #3 (HIGH): Preload After S04 Produces Incoherent Results → RESOLVED

The outside voice correctly identified that preloading after S04 with an incomplete ForgeProfile (no crystal, no headline, no constraints) produces results that differ from the final run. The build spec proposed graduated preloading, not full pipeline.

**Solution:** Graduated preloading, not full pipeline at S04.

```
GRADUATED PRELOAD STRATEGY:
  After S04: domain_match_only = true
    - Run eliminateIdeas() with industry filter only
    - Cache the filtered set (typically 30-80 ideas from 260)
    - Do NOT run full scoring (incomplete profile)
  
  After S06 (crystal): profile is mostly complete
    - Run full pipeline on the cached filtered set
    - Store preliminary results
  
  After S07 (constraints): profile is COMPLETE
    - Re-run pipeline with final constraints
    - This is the definitive result
    - If S07 result matches S06 preliminary, instant reveal at S08
    - If different, re-score during Forge animation
```

### OV #9 (LOW): Timing Singleton Clobbers Concurrent Interactions → RESOLVED

`lib/timing.ts` has a single `_startTime` variable. Multiple concurrent timers (blots, industry dwell) will clobber.

**Fix:** Return a timer instance from `startTimer()` instead of using module-level state:
```typescript
function createTimer() {
  let _start = 0;
  return {
    start: () => { _start = performance.now(); },
    stop: () => performance.now() - _start,
    peek: () => performance.now() - _start,
  };
}
```

Callers create their own timer instances. No shared mutable state.

### USER #1: S09 Triple Fallback Architecture → NEW

What happens when ideas don't load at S09? Three explicit fallback levels:

```
S09 IDEA DISPLAY FALLBACK CHAIN:
  Level 1 (happy path): 3 matched ideas from scoring pipeline
    - Display nest, spark, wildvine with full metadata
    - "Why You" text from LLM or static fallback

  Level 2 (partial match): <3 ideas after all scoring
    - Widen filters: drop time/budget constraints
    - Re-run pipeline with relaxed filters
    - If still <3: fill remaining slots with highest-scoring 
      ideas from ANY domain (labeled "Wildcard Match")

  Level 3 (catastrophic): scoring engine throws or returns 0
    - Display 3 hand-curated "safe" ideas (popular across all profiles)
    - Clearly labeled: "We had trouble finding your perfect match. 
      Here are popular ideas while we fix this."
    - Log error to Sentry with full ForgeProfile for debugging
    - Show retry button: "Try again"

  NEVER show blank. NEVER show a loading spinner forever.
  NEVER crash. Always show 3 ideas.
```

`eliminateIdeas()` in the engine requires `resource_level` and `time_budget` which don't exist at S04. The plan's "domain_match_only" flag doesn't exist.

**Fix:** Add a `filterByIndustryOnly()` function to the orchestrator (NOT the engine). This is a simple pre-filter, not the full `eliminateIdeas()`:

```typescript
// In orchestrator.ts
function filterByIndustryOnly(ideas: Idea[], industriesKept: string[]): Idea[] {
  if (industriesKept.length === 0) return ideas; // no filter if nothing kept
  const domainMap = DOMAIN_TO_INDUSTRY; // already in engine
  return ideas.filter(idea => {
    const industry = domainMap[idea.domain_primary];
    return industriesKept.includes(industry);
  });
}
```
